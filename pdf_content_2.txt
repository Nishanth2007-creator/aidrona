1. App Architecture Overview
AiDrona is a Flutter mobile application with three distinct user roles: Patient (blood requester and
donor), Doctor, and Admin. Each role sees a different navigation structure and set of pages. All
three roles share the same backend, which is a Node.js API server sitting in front of Google Cloud
Firestore and integrated with Gemini API for AI operations.

5. Gemini API Integration (Vertex AI)
All AI operations in AiDrona are handled by Gemini 1.5 Pro accessed through Google Vertex AI.
The backend Node.js server is the only component that calls Gemini. The mobile app never calls
Gemini directly.
5.1 Setup and Configuration
// Install SDK
npm install @google-cloud/vertexai
// server/ai/gemini.js
const { VertexAI } = require("@google-cloud/vertexai");
const vertex = new VertexAI({
project: "aidrona-prod",
location: "asia-south1", // Mumbai region for lowest latency in India
});
const model = vertex.getGenerativeModel({ model: "gemini-1.5-pro" });
5.2 Blood Request Triage
Called immediately when a blood request is submitted. Returns severity score, recommended

- Blood type needed: ${blood_type}

- Location: ${location.lat}, ${location.lng}
- Time of day: ${time_of_day}
Return ONLY a JSON object with:
{
"severity_score": ,
"recommended_radius_km": ,
"estimated_response_minutes": ,
"triage_reasoning": ""
}
`;
const result = await model.generateContent(prompt);
return JSON.parse(result.response.text());
}

6.3 Firestore Security Rules
rules_version = '2';
service cloud.firestore {
match /databases/{database}/documents {
// Users can only read and write their own profile
match /users/{uid} {
allow read, write: if request.auth.uid == uid;
}
// Medical records: read own, write only from server (doctor flow uses Admin SDK)
match /medical_records/{uid} {
allow read: if request.auth.uid == uid;
allow write: if false; // Server-side only via Admin SDK
}
// Crisis requests: authenticated users can create, read all open ones
match /crisis_requests/{id} {
allow read: if request.auth != null;
allow create: if request.auth != null;
allow update: if false; // Server-side only
}
// Admin collections: server-side only
match /blood_banks/{id} {
allow read: if request.auth != null;
allow write: if false;
}
}
}

AiDrona is an emergency health coordination app purpose-built for real Indian crisis scenarios —
from a dog-bite victim needing a timely rabies vaccine schedule to an accident patient urgently
requiring a rare blood type. The platform consolidates scattered information (WhatsApp groups,
phone calls, multiple apps) into a single AI-powered interface that acts within seconds.
Core Problem
When emergencies strike, critical information lives across different apps, contacts, and phone calls —
causing dangerous delays. AiDrona collapses that friction into one coordinated loop.
The AI engine (internally called AiDrona AI) performs three simultaneous tasks:
• Triage scoring — assesses severity and urgency of the incoming crisis report
• Supply anomaly detection — monitors blood bank stock and vaccine availability in real time
• Smart matching — ranks and routes requests to the most eligible responders nearby
Two primary crisis flows are supported today, with the architecture designed to
accommodate additional emergency types in future:

Medical Record Trust Layer
Reports must be signed and submitted by a registered doctor through the app. This means AiDrona is
not relying on self-reported data — the doctor's verification forms a credible trust layer that
meaningfully differentiates this platform from simple blood-donor directories.
Report expiry: Fitness scores should be re-verified every 6 months to reflect changes in health status.
After registration, a QR code is generated for the user. The QR links to a live profile (not a static
snapshot), so scanning it always shows current blood type, last donation date, hemoglobin, and
fitness status. This is primarily used by doctors on the ground when a donor arrives.

3. Blood Donor Workflow
This section describes the full end-to-end flow from a blood request being raised to a donor being
confirmed or a blood bank being contacted.
Phase 1 — Request & Triage
• User submits a blood request with blood type, current location, and urgency level.
• AiDrona AI assigns a severity score and sets an initial search radius (default: 5 km).
Phase 2 — Contacts-First Search
• The app reads the requester's phone contacts (with prior consent granted at onboarding).
• From those contacts, it filters for users who: (a) have the AiDrona app installed, (b) match the
blood type, (c) fall within the set radius, and (d) have a passing fitness score.
• The top-ranked eligible contacts receive a popup notification with accept / decline options.
• A countdown timer starts. If contacts respond before it expires, the flow proceeds to Phase 4.
Phase 3 — Fallback (User-Directed)
If no eligible contacts are found within the radius, or the timer expires with no response, AiDrona
presents the requester with two explicit options:

Stranger Notification Logic
When the 'open to strangers' path is activated:
• All eligible nearby users receive a popup with accept / decline.
• The moment one person accepts: all other popup recipients are immediately notified that the
request has been filled — they do not need to take action.
• The accepted donor receives the donation location (routed via the nearest hospital to protect the
requester's private address).
• If no one accepts within the timer window: the nearest blood bank is automatically alerted with
the blood type and urgency level, and the requester is notified that a bank has been contacted.

Phase 4 — Donor Confirmation & Verification
• Accepted donor's ETA is shown to the requester.
• On arrival, the doctor scans the donor's QR code to verify fitness in real time.
• If the doctor marks the donor as unfit: AiDrona re-triggers the search, skips this donor in the
current queue, and applies a penalty to their fitness score in the system.
• Fitness score is permanently updated after every live doctor check — the system learns from
each interaction.

5. Vaccine & Rabies Tracking Workflow
When a user reports a dog bite or animal attack, AiDrona triggers a parallel vaccine coordination
flow alongside any blood-related needs.
Step 1 — Incident report
• User submits 'dog bite' with location and date/time of the bite.
• AiDrona AI triages rabies risk based on bite severity and animal type.
• App immediately shows nearby hospitals with live rabies vaccine stock.
Step 2 — Dose 1 & schedule creation
• User selects a hospital and logs Dose 1.
• AiDrona auto-creates a full dosage schedule using the standard post-exposure protocol:

Step 3 — Pre-visit stock check
• Before each due date, AiDrona checks the scheduled hospital's vaccine stock in real time.
• If stock is low (< 2 doses) or zero, the app identifies the nearest alternative hospital with
availability and updates the dose record automatically.
• User is notified of the change before they travel — eliminating wasted trips to empty hospitals.
Step 4 — Check-in & passive stock tracking
• After receiving each dose, the user checks in via the app.
• This check-in passively decrements the hospital's vaccine stock count — so even if hospital
staff forget to manually update their inventory, real usage data keeps the stock numbers
accurate.
Vaccine Stock Data Strategy

7. AI Engine Logic
AiDrona AI is not making arbitrary decisions — it reads from the database tables described in
Section 6, scores based on defined formulas, and writes results back. Every action is traceable.
Blood crisis — runtime logic
• A CRISIS_REQUEST row is created with status = 'open' and current_radius_km = 5.
• The backend queries DONOR_PROFILE joined with USERS, filtered by blood type and radius,
ordered by fitness_score DESC.
• Top 3–5 donors from the requester's contacts receive a Firebase push notification.
• A background job checks every 2 minutes — if no acceptance, current_radius_km is
incremented and the query re-runs.
• On acceptance: DONOR_RESPONSE row updated to 'accepted'; all other pending rows
updated to 'filled'; location of receiving hospital sent to the accepted donor.
• On blood bank escalation: nearest HOSPITAL with non-zero VACCINE_STOCK (or blood
bank) is queried and alerted via API/SMS; requester notified.
Vaccine schedule — runtime logic
• One VACCINE_SCHEDULE row is created on Day 0.
• Five DOSE_LOG rows are auto-inserted with pre-calculated due_date values.
• 48 hours before each due date, the system queries VACCINE_STOCK for the assigned
hospital.
• If doses_available < 2: nearby hospitals are ranked by distance and availability; the
DOSE_LOG hospital_id is updated; user is notified.
• On check-in: taken_date is filled, VACCINE_STOCK.doses_available is decremented by 1,
status set to 'completed'.
AI Prompt Structure (Gemini / AiDrona AI)
Input to AI:
{ "incident_type": "blood_request", "blood_type": "O+", "severity": "critical",
"location": [lat, lng], "donor_candidates": [{ "donor_id": "...", "fitness_score":
87, "distance_km": 1.2, "last_donated": "2024-09-10" }, ...] }
Expected AI output:
{ "triage_score": 9.2, "ranked_donors": ["donor_id_1", "donor_id_3", "donor_id_2"],
"expand_radius_if_no_response_mins": 8, "escalate_to_bank": true }

10. Impact & Problem Scale
AiDrona addresses two of India's most avoidable public health emergencies:

What makes AiDrona different
• Trust-first design: contacts are contacted before strangers — increasing response rates through
social familiarity.
• Doctor-verified medical records: fitness scoring is not self-reported, making donor ranking
genuinely reliable.
• Self-healing data: user check-ins passively correct stock data without depending on manual
hospital updates.
• No dead ends: every failure state (no contacts, no strangers, no donors) has a defined next step
built into the AI loop.
• QR-based live profiles: doctors can verify donors instantly on arrival without manual data entry.
